# TrainingJournal.me — design spec

**Date:** 2026-07-10 · **Status:** approved design, pre-implementation
**Origin:** generalization of the author's private journal repo and running
dashboard (a real, daily-synced instance) into a publishable library.

## 1. Pitch

Fitness data today is scattered across apps whose plans are generic and
autogenerated — trackers collect data that planners never read. With AI
assistants and MCP servers, all of a user's data can be pulled to their own
machine, where their stated goals and their actual history argue with each
other to produce plans that listen. TrainingJournal.me is the harness for
that: a template repo that turns any AI coding assistant into a personal
training-journal keeper and planner, **local-first** — core information
gathering, storage, and planning happen on the user's machine.

**Tagline: fully customisable — bound only by what can be read from your
devices.** There is no fixed metric list, no fixed sport, no fixed plan
shape: if a source can be read (MCP server, API, even a file export), it
can feed the journal, and the plans adapt to whatever is actually there.

## 2. Form and distribution

- A **public GitHub template repo** `trainingjournal-me` ("Use this
  template" → private copy → clone → open in an AI harness → `/initiate`).
- **Claude Code first-class** (skills, MCP setup automation), with
  `AGENTS.md` describing the same flows as prose for other harnesses
  (Cursor, Codex, etc.).
- License: MIT. The template ships **zero personal data** — skeleton files
  with `(fill in)` markers and one fictional example profile in docs.
- Not in v1: npm scaffolder (`create-training-journal`) — a v2 distribution
  channel once the template stabilizes.

## 3. Architecture

Three trust zones:

1. **Fitness services (cloud):** Garmin, Strava, Fitbit, etc. — reached only
   through MCP servers configured on the user's machine.
2. **The user's machine (local-first, source of truth):** the private clone —
   `journal/` markdown, the AI harness with its skills, MCP servers, and the
   static build step that produces a secret-free data snapshot.
3. **Public web (zero credentials):** a static dashboard on Vercel rendering
   the snapshot, plus an **optional** Supabase project acting as a transient
   inbox for browser-side notebook entries.

Data flows: services → MCP → harness → `journal/` (pull);
user ↔ harness (goals, check-ins, plans); `journal/` → snapshot → Vercel
(deploy); browser notebook → Supabase inbox → drained back into `journal/`
on the next sync (optional loop).

## 4. Repo layout

```
journal/                    the private data (skeletons in the template)
  profile.md                who I am
  history.md                training history, PRs, injuries
  training.md               current actual routine — splits, sessions,
                            knowledge level, prescription preferences
  goals.md                  stated goals, races, target times
  gear.md                   shoes, watch, apps
  health.md                 baselines (RHR, HRV, sleep), conditions
  meals.md                  pre-saved meal library (portions, kcal, protein),
                            day templates, targets
  plans/                    generated plans, dated
  entries/YYYY-MM.md        journal — dated, newest-first, append-only
  analysis/                 sync-maintained analyses (fuel-performance lags)
  people/<slug>/…           additional people (same structure; see §9)
.claude/skills/             initiate, journal, remember, pull-data, plan, sync
AGENTS.md                   the same flows for non-Claude harnesses
app/                        static Next.js dashboard (see §6)
docs/                       conventions, privacy model, Supabase setup,
                            example profile (fictional)
CLAUDE.md                   repo rules for agent sessions
README.md                   pitch, quickstart, index
```

## 5. Skills (the harness)

All skills follow the conventions proven in the author's `E:\Me` repo:
facts only, no fabrication, `(fill in)` placeholders, dated entries,
append-only journal, subjective-vs-objective divergences recorded explicitly.

- **`/initiate`** — onboarding wizard, run once:
  1. Interview: which trackers/apps does the user use?
  2. MCP setup: configure the matching MCP servers (automated in Claude
     Code; manual walkthrough via docs elsewhere). Verify each with a test
     call.
  3. History pull: seed `journal/` from real data (activity history,
     wellness baselines) — sparse real data over dense fabricated data.
  4. **Training background → `training.md`** ("give the routine maker your
     actual workouts"):
     - Self-assessed **knowledge level** (new → experienced → advanced) —
       calibrates how much plans explain vs assume, and how aggressive
       progressions may be.
     - **Current actual split for every workout type** the user does:
       running week structure and paces, cycling, swimming, classes —
       whatever is real today, not what an app once prescribed.
     - **For gym/strength work, go deep:** do they have a preferred split
       (PPL, upper/lower, full-body…)? Which days, which exercises, which
       lifts are fixed anchors, known working weights? Capture the split as
       they actually run it.
     - **Prescription preference:** should strength plans carry sets, reps,
       RPE, and weight targets — or stay exercise-list-only? (Both are
       first-class; see `/plan`.)
  5. Goals: elicit races, target times, constraints → `goals.md`.
  6. Optional: cloud inbox (Supabase, §7), password gate (§8), first deploy.
- **`/journal`** — dated entry in `entries/`: what happened, how it felt,
  what the data said, decision/lesson. Promotes durable conclusions to topic
  files.
- **`/remember`** — file one durable fact in the right topic file; replace
  stale values in place; new topic → new file + index row.
- **`/pull-data`** — on-demand retrieval from connected MCPs with
  reconciliation defaults: synced copies are identical; parallel phone
  recordings read short (device/watch copy canonical — flag duplicates);
  GPS beats treadmill-mode outdoors; prefer aggregate endpoints over
  intraday payloads; compare against `health.md` baselines.
- **`/meal`** — fuel tracking as a training signal. Logs meals by resolving
  against the pre-saved library in `meals.md` (one-line logging for repeat
  meals; honest RANGES for unlisted ones, never fake precision). Computes
  the day's calories in (with coverage stated), expenditure (tracker active
  calories + BMR calibrated against the real weight trend; steps as a
  cross-check), and energy balance vs target. Browser-side meal logging
  rides the inbox with kind "meal". Fuel rules feed planning:
  - **surplus is fuel** — a cheat meal today marks tomorrow as
    glycogen-backed and may nudge the suggested tier toward overreach,
    but never overrides readiness/HRV gates;
  - **deficits and quality sessions don't mix silently** — fueling notes
    on hard sessions during multi-day deficits; protein floor flagged;
  - **the deficit-performance lag is actively hunted**: the sync maintains
    `journal/analysis/fuel-performance.md`, pairing each day's balance
    with subsequent performance markers and testing 1-5 day lags. Recurring
    patterns (≥3 instances) surface as named hypotheses with evidence
    counts — watched and planned around, never claimed as causation.
- **`/plan`** — the plan builder: reads history + goals + `training.md`
  (split, knowledge level, prescription preference) + current baselines
  (readiness, HRV vs average, recent load), produces a dated plan in
  `journal/plans/` and updates the dashboard snapshot. Rules:
  - **Every prescribed session ships three tiers** (proven in RunPlan):
    - **underreach** — "not feeling 100": volume/intensity trimmed,
      quality stripped;
    - **target** — the baseline-goal session as designed;
    - **overreach** — "feeling 200": a genuine step up, explicitly gated on
      readiness AND recovery signals being above baseline that day, never a
      default.
  - The **suggested tier** is derived at view time from that day's
    readiness, HRV vs its average, and the user's stated feel — never
    precomputed into the data.
  - **Strength sessions may prescribe sets, reps, RPE, and weight targets**
    calibrated to the user's actual split and knowledge level — if and only
    if they opted into that in `training.md`; otherwise exercise lists with
    effort guidance.
  - Paces/loads are ceilings when recovery signals are below baseline;
    plans must cite the data they used ("your HRV avg is 46 and your goal
    is sub-2:00 — this plan assumes both"); never invent history.
- **`/sync`** — the recurring loop (manual trigger, or installed as a
  scheduled routine): **read the morning's recovery signals (HRV, sleep
  score, readiness) at routine time** and refresh the day's suggested-tier
  inputs → drain the inbox (§7) → pull fresh daily data → append real trend
  points → generate today's workout detail (three tiers, day-of only by
  default) → regenerate the snapshot → `vercel deploy`. Same pattern as the
  author's daily sync routine.

## 6. Dashboard (`app/`)

Static Next.js app deployed to Vercel with no credentials — it renders a
generated data snapshot bundled at build time.

- **Views:** trends (zoomable real-data charts, sparse-honest), current plan,
  recent activities, journal notebook, settings.
- **Tier picker:** today's session renders underreach / target / overreach
  pills with the suggested tier badged, computed in the browser from the
  snapshot's readiness/HRV data (mirrors RunPlan's `computeSuggestedTier`).
- **Snapshot format:** `{ people: [{ id, name, data }] }` — an array from
  day one (length 1 in single-user mode, §9).
- **Notebook:** a text input for journal entries from the browser. With
  Supabase connected it posts to the inbox; without it, it renders a
  "paste this to your agent" fallback. Settings includes a guided
  "Connect Supabase" option.
- Charts follow the honest-data rules: real dated points only, window-scoped
  averages, no interpolation.

## 7. Supabase — the drained inbox (optional)

Supabase is a **mailbox, not a database of record**. Steady state is empty.

- **Schema:** one table `inbox`:
  `id uuid · created_at timestamptz · person text · kind text
  ("journal" | "checkin" | "meal" | "coach-note") · payload jsonb`.
  Nothing else goes up — no profile, no history, no metrics.
- **Write path:** the dashboard notebook inserts with the anon key.
  RLS: anon may **insert only** — no select, no delete. The browser can add
  to the inbox but never read it.
- **Drain path:** `/sync` (scheduled or manual) reads all rows with the
  service key (local machine only), writes each into
  `journal/entries/YYYY-MM.md` keyed by `id` (idempotent — a crash mid-drain
  cannot duplicate), then **deletes the drained rows in the same run**.
- **Staleness surface:** the dashboard may show "N entries waiting for your
  next sync" (count via a security-definer function; still no content
  readable by anon).

## 8. Password gate

Simple username/password gate for the dashboard — asked once, hashed one
way, plaintext never stored.

- **Setup:** during `/initiate` or from settings, the user picks a
  username + password. The agent derives `PBKDF2-SHA-256(password, salt,
  ~300k iterations)` locally and writes `{username, salt, params, hash}`
  into the public snapshot config. The plaintext is never written to disk,
  journal, config, or Supabase.
- **Gate:** the dashboard renders a login screen first; the browser derives
  the same hash and compares; success sets an unlock flag in `localStorage`.
  Everything, including the notebook, sits behind the gate.
- **Documented limitation:** a client-side gate on a static site deters
  visitors and crawlers but the snapshot still ships in the bundle — it is
  a privacy curtain, not encryption. **Upgrade path (v2, documented):**
  encrypt the snapshot with a key derived from the same password; identical
  UX, real secrecy.
- One shared password per deploy (household model); per-person auth is out
  of scope for v1.

## 9. Multi-person provision

Every data structure carries a `person` id from day one so multi-user is an
array append, not a migration.

- **Tier 1 — same clone/household (v1):** default person lives at
  `journal/`; additional people at `journal/people/<slug>/` with the same
  structure. Skills resolve the active person from context or ask once.
  MCP connections are per-person. The dashboard shows a person switcher
  only when `people.length > 1`. Inbox rows carry `person`.
- **Tier 2 — separate machines, one shared dashboard (v2, designed now):**
  static bundling can't merge two machines' data (last deploy wins), so
  shared mode moves the **snapshot** (and only the snapshot — the same
  secret-free data that would have been bundled) into a Supabase
  `snapshots` table, one row per person, overwritten each sync; the
  dashboard reads snapshots at load time. Raw journals, credentials, and
  MCP access never leave each member's machine.
- **Coach mode (freebie):** anyone with dashboard access (behind the gate)
  can drop a `kind: "coach-note"` inbox entry addressed to a person; it
  lands in that person's journal at their next sync for their agent to
  weigh.

## 10. Privacy model (one paragraph)

Identity and history exist only in the private clone on the user's machine.
Vercel receives a static, secret-free snapshot. Supabase — if enabled at
all — holds an insert-only inbox drained to zero on every sync, plus (in
Tier-2 shared mode only) per-person snapshots equivalent to what the static
bundle would have contained. The template repo itself contains no personal
data. The password gate deters casual access and its limits are documented
honestly.

## 11. Testing & verification

- **Skills:** each runs against a fixture `journal/` (fictional person) in
  dry-run mode; assertions on file outputs. The reconciliation rules in
  `/pull-data` get fixture cases (duplicate recording, treadmill vs GPS).
- **Inbox drain:** idempotency test — drain the same rows twice, entries
  appear once; crash-mid-drain simulation leaves no loss and no duplicates.
- **Password gate:** hash derivation cross-checked agent-side vs
  browser-side; wrong password renders nothing.
- **Dashboard:** the RunPlan verification pattern — production build, DOM
  overflow audit at 375px and desktop, NaN audit on all SVG charts, both
  color schemes.

## 12. Connector landscape (surveyed 2026-07)

What `/initiate` can wire up today. Four tiers, best experience first:

**Official MCP connectors (first-party, plug-and-play):**

| Tool | Notes |
|---|---|
| Strava | Official MCP connector since Jun 2026 — subscriber-only, read-only, revocable; activity history, fitness trends, readiness, gear. Community MCPs also exist (r-huijts/strava-mcp and others). |
| Runalyze | MCP over its aggregation of Garmin, Polar, Suunto, COROS, Whoop, Oura + health layer (Jun 2026). |

**Community MCP servers over official APIs (API key/OAuth, self-hosted):**

| Tool | Notes |
|---|---|
| Garmin | No official MCP; mature community servers (e.g. `garmin_mcp`, ~96 tools) — the one this design was proven on. |
| Whoop | `whoop-mcp` (~18 endpoints: cycles, recovery, strain, workouts). Caveat: Strength Trainer workout data is NOT exposed by Whoop's API. |
| Oura | `oura-mcp` (~20 tools: sleep, readiness, resilience) over API v2 token. |
| Fitbit / Google Health | Community MCPs for the Fitbit Web API and its successor Google Health API v4 (migration through 2026 — OAuth/scopes/endpoints changing; recipes must track it). |
| Hevy | `hevy-mcp` (TypeScript + Rust variants): full workout/routine/exercise CRUD — the strongest strength-logging source. Requires Hevy PRO API key. |
| Intervals.icu | Community MCP + present in aggregators; strong training-analysis layer. |

**Aggregator MCPs (one connection, many sources):**

| Tool | Notes |
|---|---|
| athletedata.health | Hosted MCP exposing 20+ sources (Strava, Garmin, Whoop, Oura, Hevy, Withings, Apple Health…). Convenient but hosted — data flows through a third party, which cuts against local-first; document the trade-off. |
| Open Wearables | Open-source multi-platform server (~6 platforms). |
| Runalyze | (also listed above) doubles as an aggregator for Polar/Suunto/COROS users. |

**API-only or export-only (bridgeable, no turnkey MCP):**

| Tool | Route |
|---|---|
| Apple Health | No cloud API — on-device HealthKit only. Bridges: Health Auto Export-style apps, third-party MCP bridges, or import into Google Health and read from there. |
| Polar / Suunto / COROS | Official APIs (AccessLink etc.); easiest path today is via Runalyze/aggregators. |
| Withings | Official API; available via aggregators — body composition/weight. |
| TrainingPeaks | Partner API (approval required). |
| Samsung Health | Health Connect on Android; export tools. |
| Runna | No public API — but its plans sync into Garmin as structured workouts, which is exactly how this design reads them (proven pattern). |
| Strong, Peloton, MyFitnessPal | CSV export only / unofficial API / restricted partner API — file-import recipes at best. |

v1 ships recipes for Garmin, Strava (official connector), and Hevy;
Whoop/Oura/Fitbit-Google recipes are v1-if-time, the rest documented
pointers. The landscape moves monthly — `docs/sources.md` in the template
is the living version of this table.

## 13. v1 scope vs v2

**v1:** template repo, seven skills (incl. meal/fuel tracking with the
lag-correlation hunt), dashboard (trends/plan/notebook/settings),
Supabase inbox + drain, password gate, Tier-1 multi-person, AGENTS.md,
docs with fictional example.
**v2:** encrypted snapshot, Tier-2 shared dashboards, npm scaffolder,
scheduled-sync installers for more platforms, more MCP source recipes
(Fitbit/Google Health, Apple Health, Oura).
